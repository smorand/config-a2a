# A2A protocol surface

The endpoints below are what `web-a2a` (or any A2A v1.0 client) hits. Routes live in `src/config_a2a/a2a/routes.py`; the wire model is in `src/config_a2a/a2a/envelope.py`.

## Discovery

```
GET /.well-known/a2a/agent-card        # canonical
GET /.well-known/agent-card.json       # alias for older clients
```

Returns the Agent Card with `capabilities.streaming: true`, the configured `skills`, and (when authentication is on) `securitySchemes` + `security`.

## Messaging

```
POST /message:send                    # synchronous, returns final Task JSON
POST /message:stream                  # SSE; event types: task, statusUpdate, artifactUpdate
GET  /tasks/{id}                      # fetch the persisted state
POST /tasks/{id}:cancel               # request cancellation
GET  /tasks                           # list the most recent (capped at 100)
GET  /health                          # liveness, no auth
```

### SendMessage payload

```jsonc
{
  "message": {
    "messageId": "<uuid>",
    "role": "ROLE_USER",
    "contextId": "<uuid>",            // optional — autogenerated if missing
    "taskId": "<uuid>",               // present only when resuming a paused task
    "parts": [{ "text": "..." }]
  }
}
```

### Resume

If `message.taskId` is set, the router looks up the existing task. Unknown ids return 404. The runtime appends the new message to the task history before running the pattern, which lets `simple.py` detect a pending `confirm_tool` action and continue from there.

### Streaming events

```jsonc
{ "task": { "id": "...", "contextId": "...", "status": {...} } }
{ "statusUpdate": { "taskId": "...", "contextId": "...", "status": {...}, "final": true|false, "metadata": {...} } }
```

`statusUpdate.status.state` is one of:

- `TASK_STATE_SUBMITTED` (initial)
- `TASK_STATE_WORKING`
- `TASK_STATE_INPUT_REQUIRED` (paused for user confirmation; `final: false`)
- `TASK_STATE_COMPLETED`
- `TASK_STATE_FAILED`
- `TASK_STATE_CANCELED`

### INPUT_REQUIRED for destructive tools

When the model calls a tool with `annotations.destructiveHint=true` and `confirmations.destructive_hint` is `prompt`, the runtime emits:

```jsonc
{
  "statusUpdate": {
    "taskId": "...",
    "contextId": "...",
    "status": {
      "state": "TASK_STATE_INPUT_REQUIRED",
      "message": { "role": "ROLE_AGENT", "parts": [{ "text": "About to call destructive tool..." }] }
    },
    "final": false,
    "metadata": {
      "kind": "confirm_tool",
      "tool_name": "fs.delete_file",
      "arguments": { "path": "/tmp/x" },
      "tool_call_id": "..."
    }
  }
}
```

The task is persisted with `pending_action = <metadata>`. The user resumes with a new message (same `taskId`); replying `yes`/`approve`/`ok` continues, anything else cancels.
